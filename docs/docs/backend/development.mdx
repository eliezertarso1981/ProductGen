# Backend

O backend fica em `api/`.

## Rodar localmente

```bash
cd api
npm install
npm run dev
```

Por padrão a API sobe em:

```text
http://localhost:3000
```

Health check:

```text
GET /health
```

OpenAPI JSON da API:

```text
GET /openapi.json
```

## Módulos implementados

- `auth`: login local com senha, access token curto em cookie HTTP-only e sessão renovável.
- `products`: CRUD de produtos por workspace.
- `workspace-members`: CRUD de membros do workspace.
- `evidences`: CRUD e transição de status de evidências.
- `insights`: CRUD de insights e vínculo com evidências.
- `pains`: CRUD, transição de status, merge/split e scoring (RICE/ICE).
- `hypotheses`: CRUD, transição de status, clone e scoring (RICE/ICE).
- `experiments`: CRUD e transição de status/resultado de experimentos.
- `links`: vínculos dor↔hipótese e hipótese↔roadmap.
- `roadmap`: CRUD, transição de status e scoring (RICE/ICE).
- `strategic-pillars`: CRUD de pilares estratégicos por produto.
- `objectives`: CRUD de objetivos (OKR) e transição de status.
- `key-results`: CRUD de key results por objetivo.
- `outcomes`: CRUD de outcomes por item de roadmap e transição de status.
- `prds`: PRDs versionados por item de roadmap.
- `releases`: CRUD de releases por produto.
- `engineering-handoffs`: handoff de roadmap para Jira/ADO.
- `platform`: comentários, atribuições, decision logs, leitura de eventos e assets polimórficos.
- `analytics`: endpoints sobre views consolidadas.

## Exportar e validar OpenAPI

```bash
cd api
npm run openapi:export
```

O arquivo gerado vai para:

```text
docs/static/openapi/productgen.json
```

Antes de abrir PR, confira se o contrato está sincronizado:

```bash
cd api
npm run openapi:check
```

## Estratégia de autenticação

O ProductGen usa autenticação local por enquanto, mas a API isola esse comportamento por trás de uma fronteira de provedor de identidade. Isso mantém `users`, `workspaces` e `workspace_members` como fonte de autorização do produto e deixa espaço para trocar o provedor de login no futuro.

Fluxo atual:

1. `POST /auth/login` valida `email` e `password`.
2. A API grava `pg_access` e `pg_refresh` como cookies HTTP-only.
3. O access token é RS256 e contém apenas `user_id`, `email` e `session_id`.
4. O refresh token é opaco, salvo como hash em `user_sessions` e rotacionado em `POST /auth/refresh`.
5. Rotas protegidas resolvem o workspace ativo por `X-Workspace-Id`, cookie `pg_workspace` ou último acesso.
6. `GET /auth/me` faz o bootstrap do usuário, workspaces e produtos acessíveis.
7. `POST /auth/logout` revoga a sessão atual e remove os cookies.

Keycloak ou outro provedor OIDC deve ser adotado quando houver demanda concreta de SSO corporativo, SAML/LDAP/AD, self-host por compliance ou capacidade operacional para manter realms, clients, upgrades, backup e observabilidade.
